## 2. 问题清单 & 根因分析

### 问题1：pyMuPDF vs pdfminer.six 选择失误

**现象**：20篇病证治法PDF被pdfminer判定为"截图/零文字"，浪费整整一批次时只插入了文件名推断的内容，后来发现PyMuPDF(fitz)能正常提取。

**原因**：
- pdfminer.six 依赖 PDF 的显式文本层（Tj/TJ 操作符），遇复杂编码或非标准PDF解析策略则返回空白
- PyMuPDF(fitz) 使用 MuPDF 引擎，容错更强，能处理更多PDF变体
- 决策时基于"brew编译太慢→装不了poppler"的逻辑跳过了更关键的"测试多个PDF引擎"

**教训**：**PDF提取工具的选型必须对样本集做全量测试**，而非一个样本就下结论。

### 问题2：macOS `/tmp` 不是持久存储

**现象**：`fix_fw.py` 脚本写在 `/tmp/` 路径，macOS重启/长时间后因tmp清理机制丢失。

**原因**：OS X 的 `/tmp` 被视作临时空间，系统定期清理。脚本应该写在 workspace 内。

**教训**：**所有文件必须写在 `~/.openclaw/workspace/` 下**，不依赖任何系统临时目录。

### 问题3：记忆Flush回合限制文件写入

**现象**：教学群问答的Python插入脚本写好但无法执行，必须等到下一个"非memory flush"回合。

**原因**：Runtime event system 的约束——某些回合只允许写 memory/ 文件，不能修改框架。

**教训**：**写框架/改系统文件的大操作应该在"非memory flush"回合集中执行**，memory flush回合只做状态记录。

### 问题4：待处理清单存在漏匹配（全角/半角括号）

**现象**：待处理清单中 金匮系列只列出9篇，但实际目录有51篇；部分文件名因全角括号 `《》` vs 半角 `()` 导致匹配失败。

**原因**：构建待处理清单时用 date prefix 匹配，忽略了文件名格式差异；源索引文件名也存在全角/半角不一致。

**教训**：**文件名匹配必须做名称归一化**，匹配逻辑要用更robust的近似匹配（去掉标点符号差异）。

### 问题5：Git push间歇性失败（中国网络环境）

**现象**：多次 `git push origin main` 因 Cannot connect / HTTP2 framing error 失败，有时重试后成功，有时需要改代理。

**原因**：
- GitHub在中国大陆的访问受网络限制
- ClashX 代理未运行或未配置成功时，直连超时
- HTTPS 的 HTTP2 协议遭遇中间网络干扰

**教训**：**Git push 应预留重试机制和重试间隔**，明确代理状态。

### 问题6：229篇"Z-其他专题"的虚假工作量

**现象**：一开始以为有229篇待处理，但实际大部分是单药文章（已通过7.67方药汤方表格覆盖），最终实际上只剩5篇真正未处理。

**原因**：待处理清单构建时只匹配了文件名，没有检查内容是否已被框架覆盖。单药文章虽然文件名不同但内容已汇总在7.67中。

**教训**：**"待处理"数量和"真正需要处理"数量之间要做框架内容级去重**，不能只做文件名级去重。

### 问题7：金匮系列重复索引

**现象**：框架原已有"金匮线56"（56篇金匮PDF已索引），但待处理清单又列出9篇"金匮注解"→实际是同一批文件，误以为需要处理。

**原因**：待处理清单构建时，源索引中的"金匮线56"使用了不同的文件名格式，导致9篇被误判为"新文件"。

**教训**：**双方向匹配验证**——既从文件名角度查是否在源索引，也从源索引反查文件是否存在。

## 3. 工作规范（Standard Operating Procedure）

### 3.1 准备阶段

```
[ ] 确认 workspace 路径（~/.openclaw/workspace 下，非 /tmp）
[ ] 检查目标仓库的可访问性（测试 git push）
[ ] 确认代理状态：git config http.proxy http://127.0.0.1:7890
[ ] 确认 Python 环境和所需库（fitz/pdfminer.six 两个都装，互为备份）
[ ] 读取当前框架版本号 + 源索引总数
```

### 3.2 收集阶段

```
[ ] 列出源目录所有文件 → 统计总数
[ ] 提取所有文件名 → 统一归一化（全角→半角，去除标点差异）
[ ] 与当前框架源索引做双方向匹配
    - 文件名→框架（检查是否已索引）
    - 框架→文件名（反向查是否有同名文件）
[ ] 生成真正的"待处理清单"（排除已索引 + 排除确认跳过文件）
[ ] 分类标注排除原因（重复/通知/目录/课程/已覆盖）
```

### 3.3 分析阶段

```
[ ] 对每个待处理文件，用 pdfminer.six 提取文本
    - 若返回空 → 用 PyMuPDF(fitz) 备用方案
    - 若仍返回空 → 标记为"图像PDF，需OCR"
[ ] 提取核心论点（每篇5-8个关键句）
[ ] 归类主题（按六经取向/方剂/药物/专题）
[ ] 标注与现有框架的关联（新增观点/补充证据/颠覆性解读）
```

### 3.4 整理阶段

```
[ ] 确定插入位置（按框架结构，7.6x 新节 vs 现有章节扩写）
[ ] 分类写法：
    - 表格类：源索引表（编号+主题+核心论点）
    - 正文类：提炼精华（每模块3-5个子标题）
    - 只索引类：仅有表头的快速覆盖（如图像PDF）
[ ] 估算行数/复杂度 → 决定：单批 or 分多批
```

### 3.5 处理阶段

```
[ ] 写Python脚本（非memory flush回合）：
    1. 读框架
    2. 精确锚点定位（用grep确认锚点唯一性）
    3. 插入+计数更新
    4. 验证输出（行数/版本号/章节号）
[ ] 执行脚本 → 检查返回结果（Success/Fail 都要核实）
[ ] git add + commit（Conventional Commits 风格）
[ ] git push（若失败：重试3次，间隔10s，必要时重新检查代理）
```

### 3.6 融合阶段

```
[ ] 交叉验证：检查新插入的源索引条目的文件名是否在源目录中存在
[ ] 如果框架中有引用"本地文件"，确认无需文件即可独立使用
[ ] 更新 HOME.md / 相关索引文件
[ ] 如有Obsidian大医传承 → 创建或更新对应的源笔记
[ ] 附录到 daily memory（记录批次、数量、关键决策）
```

### 3.7 验收清单

```
[ ] 版本号已更新
[ ] 源索引总数正确（old + new）
[ ] 框架行数无异常增减
[ ] 每个新章节内容可独立阅读
[ ] git commit 存在且有意义的message
[ ] git push 成功（或已注释pending）
[ ] 不依赖本地文件可访问
[ ] memory已更新
```

## 4. 决策规则

| 场景 | 规则 |
|------|------|
| 工具选择 | 并行测试两个引擎（pdfminer + fitz），选提取更多的那个 |
| 文件写入路径 | 永远在 `~/.openclaw/workspace/` 下 |
| 修改框架时机 | 只在非 memory flush 回合 |
| 单批处理量 | ≤50篇/批（经验表明30篇左右最优） |
| 锚点定位 | 必须用 grep 验证锚点在框架中唯一存在 |
| Git push | 失败后重试3次，每次间隔10s。第4次挂起并记录 |
| 待处理清单构建 | 双向校验（文件名/源索引交叉引用） |
| 内容去重 | 不仅要文件名去重，还要主题级去重（看看内容是否已通过其他文件覆盖） |

## 6. 模板脚本

```python
#!/usr/bin/env python3
"""统一数据插入模板

用法：编辑下列变量，运行即可
"""

FW = "skills/tcm-knowledge/references/models/dawn-su-wen-framework.md"

# === 配置区 ===
BATCH_NAME = "批次名"        # 如 "方药/汤方"
BATCH_SIZE = 30              # 新增PDF数
ANCHOR = "上一个章节的锚点"   # 用 grep 确认唯一
COUNT_OLD = 421              # 旧总数
VERSION = "v2.4"

# 源索引表（多行字符串）
SI_TABLE = """
### 源索引表

| 编号 | ...
"""

# 正文
CONTENT = """
#### 7.xx 标题
...
"""

# === 执行（勿改）===
def insert():
    import re
    with open(FW) as f:
        text = f.read()
    assert text.count(ANCHOR) == 1, f"锚点 '{ANCHOR}' 不唯一!"
    assert str(COUNT_OLD) in text, f"旧数量 {COUNT_OLD} 未找到!"
    
    # 先插入正文
    pos = text.find(ANCHOR)
    text = text[:pos] + CONTENT + "\n\n" + text[pos:]
    
    # 再插入源索引
    si_pos = text.find("来源索引共收录")
    text = text[:si_pos] + SI_TABLE + "\n\n" + text[si_pos:]
    
    # 更新计数
    text = text.replace(f"来源索引共收录{COUNT_OLD}篇", f"来源索引共收录{COUNT_OLD+BATCH_SIZE}篇")
    
    with open(FW, "w") as f:
        f.write(text)
    print(f"✅ 完成！行数={len(text.split(chr(10)))}")

insert()
```
